# 06. 数据、本地化、节点与测试

本章说明怎样在编辑器中维护通用数据、本地化、公共函数和节点流程，并在测试运行中确认结果。

## 通用数据

`通用数据` 用来维护不直接放在地图上的规则数据。当前模板包含：

- `Item`：钥匙、怪物手册、楼层传送器、破墙、清墙、破熔岩、破冰等道具。
- `Equip`：装备数据。
- `Enemy`：敌人名称、描述、属性、掉落和特殊效果。
- `State`：中毒、虚弱等状态，以及状态事件。
- `Class`：职业或角色分类数据。
- `Special`：模板自定义的特殊标记。

字段类型应按数据用途选择。文本使用 `string`，数字使用 `int` 或 `float`，开关使用 `bool`，资源引用使用 `file`，多值内容使用 `list` 或 `dict`。

资源字段会限制在 `Assets` 下选择文件，便于保持路径稳定。

## 通用数据事件

通用数据成员也可以有事件图。例如：

- 物品的 `onUse`。
- 状态的 `onWalk` 和 `onHookTriggered`。
- 敌人的 `onDefeat` 和 `onEncounter`。

这些事件适合放置与数据本身绑定的逻辑。比如道具使用效果放在物品数据上，状态每步触发的效果放在状态数据上。

## 公共函数

`公共函数` 位于 `Data/CommonFunctions`。当前模板包含 `PrintTest`。

公共函数适合放置可复用的节点流程，再由蓝图、物品、状态或其他节点调用。若一段节点逻辑会被多个事件重复使用，应优先整理为公共函数。

## 节点函数

节点函数按分组显示。模板常用分组包括：

- `System`
- `Scene`
- `Player`
- `State`
- `Movement`
- `Mota`
- `Math`
- `String`
- `Container`
- `Utils`

编辑节点图时，先放置能提供当前场景、玩家、地图或目标 actor 的节点，再连接要执行的功能节点。

带有多个执行结果的节点会显示多个输出分支。需要等待移动、转场、对话或选项完成的节点会在流程中暂停，完成后从对应分支继续执行。

## 本地化

项目文本可以写成本地化 key，例如 `{KEY_Y}`、`{SLIME}`。这些 key 由 `Data/Locale/Locale.xlsx` 维护，并导出为 `Data/Locale/en_GB` 和 `Data/Locale/zh_CN`。

需要改文本时，优先在本地化表中添加或修改 ID 与各语言列，然后使用编辑器中的本地化导出功能重新生成语言文件。

通用数据、蓝图和节点参数里引用本地化文本时，保持 `{ID}` 格式。

## 测试运行

保存项目后使用 `测试游戏` 启动当前项目。测试时重点确认：

- 新游戏是否从期望地图和起点开始。
- 地图图块、自动图块、光源和 actor 是否显示正确。
- 碰撞、重叠、门、楼梯和道具事件是否触发。
- 对话、选项、商店、天气、闪屏、震屏和转场是否按节点流程执行。
- 语言、音量、窗口皮肤、光标和标题音乐是否来自当前配置。

如果测试结果仍旧显示旧内容，先确认相关地图、蓝图、通用数据、配置和本地化表已经保存或导出。
