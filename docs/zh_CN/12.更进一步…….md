 # Ludork：2D RPG 工具包 - 帮助手册
 ## 更进一步……
 
 本章汇总一些偏“开发者向”的内容：如何扩展编辑器与运行时、性能优化思路、常见问题排查，以及许可证与致谢信息。
 
 ### 扩展编辑器
 
 #### 编辑器扩展（EditorExtensions/）
 
 Ludork 支持从 `EditorExtensions/` 目录加载原生扩展模块。这类模块通常是编译生成的 Python 扩展（例如 `.so` / `.pyd`），用于加速编辑器侧的热点工作。
 
 一个典型例子是地图编辑时的图块渲染：编辑器会优先尝试导入原生渲染函数；如果扩展不可用，则回退到纯 Python 方案继续工作，但在大地图或多图层情况下会明显更慢。
 
 实用要点：
 - 编辑器侧扩展的源码通常位于 `C_Extensions/Editor/`。
 - 构建流程一般会生成编译产物与对应的 `.pyi` 类型桩文件，并将它们放入 `EditorExtensions/`。
 - 如果扩展无法导入，编辑器不一定会不可用，但地图预览与绘制性能会下降。
 
 #### 添加自定义控件 / 面板（Widgets/）
 
 编辑器 UI 基于 PyQt5，主要代码位于 `Widgets/`，复用组件位于 `Widgets/Utils/`。添加新控件或面板时，建议：
 - 遵循现有命名与组织方式（例如 `W_*.py`、`WU_*.py`）。
 - 保持职责单一（一个面板解决一个问题）。
 - 尽量通过主窗口的菜单/工具栏/面板区域进行集成，避免无组织地新增独立窗口。
 
 设计一个新面板时，重点考虑：
 - 入口与交互流程（用户如何打开面板，如何与当前地图/图块集/脚本选择联动）。
 - 数据归属（面板编辑哪些数据、何时写回磁盘）。
 - 刷新与失效（编辑后需要刷新哪些预览或缓存，保证运行时与编辑器一致）。
 
 ### 扩展运行时
 
 Ludork 的运行时以 Python 为主，并通过原生模块进行加速。项目参考实现位于 `Sample/Engine/`，并配套了一组用 pybind11 构建的 C/C++ 扩展。
 
 #### C 扩展模块（C_Extensions/）
 
 原生模块按职责组织：
 - `C_Extensions/Editor/`：编辑器侧加速（例如编辑器中的图块渲染）。
 - `C_Extensions/GamePlay/`：玩法侧热点（例如寻路、通行缓存重建、粒子更新等）。
 - `C_Extensions/Graphics/`：图形相关工具（例如 tilemap/map 绘制辅助、动画压缩等）。
 - `C_Extensions/Utils/`：通用加速工具。
 
 这些模块通常会：
 - 向 Python 暴露可调用 API（编辑器/运行时会直接 import 并调用）。
 - 通过 `pybind11-stubgen` 生成 `.pyi`，提升类型提示与 IDE 体验。
 
 #### 通过 pybind11 添加新 C++ 模块（高层说明）
 
 新增原生模块时，建议保持结构和现有模块一致，并把 C++ 作为“加速层”而非“逻辑层”：
 - 在 `C_Extensions/` 下创建新目录，参考现有布局（`include/`、`src/`、`CMakeLists.txt`、`main.cpp`、`process.py`）。
 - 在 `CMakeLists.txt` 中使用 `pybind11_add_module(...)`，在 `main.cpp` 中暴露一个小而稳定的 Python API。
 - 参考现有 `process.py`：构建后生成 `.pyi`，并将编译产物与 `.pyi` 移动到目标目录（`EditorExtensions/` 或 `Sample/Engine/` 下对应位置）。
 
 不建议在帮助文档中写成“完整 C++ 教程”。更推荐保持接口清晰：Python 负责逻辑编排，C++ 负责确定的热点计算与批处理。
 
 ### 性能建议
 
 #### Tilemap 优化思路
 
 - 编辑器侧尽量启用原生 tilemap 渲染扩展；如果扩展缺失或加载失败，地图绘制会回退到 Python 渲染，处理大地图会变慢。
 - 降低无谓刷新：只显示正在编辑的图层，避免在同一张超大地图里堆叠过多内容（能拆分就拆分）。
 - 控制贴图尺寸：过大的 tileset 图会提升内存占用，并增加编辑器侧像素格式转换成本。
 
 #### 粒子优化思路
 
 - 控制粒子数量与发射频率；每帧创建过多短生命周期对象会造成明显的性能波动。
 - 尽可能复用/池化粒子对象，减少频繁分配与回收。
 - 如果出现卡顿，优先减少每帧的计算量：简化粒子行为、缩短生命周期、降低叠加混合层数。
 
 #### 减少内存占用建议（psutil / pympler）
 
 Ludork 在运行时调试模式下可以统计性能与内存信息，常见指标包括：
 - 进程 RSS 内存（通过 `psutil`）
 - 对象大小粗略估算（通过 `pympler.asizeof`）
 
 你可以用这些数字观察“增长趋势”，定位是否存在资源缓存只增不减、场景对象未释放等问题。常见的降内存手段包括：
 - 避免把不再使用的纹理/音频长期留在缓存中。
 - 优先使用更小的贴图与合适的音频编码格式。
 - 复用大型场景对象，减少重复创建。
 
 ### FAQ & Troubleshooting
 
 #### 常见安装问题
 
 - Python 依赖缺失：对照 `requirements.txt` 安装依赖。
 - C 扩展编译失败：确认本机有 C/C++ 编译工具链、CMake，并且构建脚本所需的 SFML 源码/路径可用。
 - Qt 平台插件问题：PyQt5 在某些环境下会因为平台插件缺失导致无法启动，通常会显示 “Qt platform plugin” 相关错误。
 
 #### 编辑器无法启动 / 崩溃
 
 - 建议从命令行启动以获得完整 traceback，优先定位第一个失败的 import/初始化步骤。
 - 如果崩溃发生在导入原生模块时，重新构建对应的 C 扩展，并确认产物被放到正确目标目录。
 - 如果仅打包后的应用崩溃，检查打包包含的 Qt 插件与 Python 模块是否齐全。
 
 #### 资源缺失 / 路径不正确
 
 - 确认项目结构完整（`Data/`、`Assets/` 及其子目录）。
 - 检查配置（例如 `Data/Configs/System.json`）里引用的文件是否真实存在。
 - 跨平台迁移项目时注意文件名大小写一致性与路径拼写。
 
 #### 运行时没有体现编辑器改动
 
 - 确认运行/预览前已保存内容。
 - 如果改动涉及会被缓存的“类代码数据”（例如公共函数、导入的模块），建议重启预览/运行会话。
 - 如果你维护了自定义加载器，确保在重载时失效缓存并重新读取最新的 JSON/数据文件。
 
 #### 打包问题（资源未打入、启动报错）
 
 - 确保打包结果包含项目数据与资源目录；缺失项目目录（例如 `Sample/` 或等价的 Project 目录）是最常见的失败原因。
 - 如果应用能启动但资源缺失，检查资源是否放在了打包脚本未复制的目录之外。
 - 如果应用无法启动，核对打包时是否包含必要模块（例如 PyQt5 的 SVG 支持）以及 Qt 插件。
