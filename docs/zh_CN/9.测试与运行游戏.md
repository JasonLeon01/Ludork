 # Ludork：2D RPG 工具包 - 帮助手册
 ## 测试与运行游戏

 本章介绍如何在 Ludork 编辑器中测试与运行游戏。目标是帮助你在“编辑内容”和“查看运行效果”之间建立顺畅的循环。

 ### 在编辑器中预览

 编辑器提供内置的游戏预览能力，你可以在不离开工具的情况下运行游戏：
 - 主窗口中的**游戏视图**可以显示接近运行时效果的画面。
 - 工具栏按钮或菜单中的“播放/预览”命令会从指定入口点启动游戏。

 典型工作流是：
 - 在编辑器中修改地图、图块集、公共函数和配置。
 - 保存修改。
 - 使用预览命令在编辑器内运行游戏。
 - 观察运行效果，然后返回编辑器继续调整。

 由于编辑器与运行时共享同一套数据格式，只要保存完成，通常就可以立即测试你的改动。

 ### 选择起始位置

 在大多数项目中，游戏会从特定配置所定义的起点开始：
 - 系统配置文件中可能定义初始地图或场景。
 - 其他配置可能指定起始队伍、摄像机位置或 UI 状态。

 当你运行或预览游戏时：
 - 运行时引擎会读取这些配置值。
 - 按照项目数据加载初始地图、角色和 UI。

 要测试游戏的不同部分，你可以：
 - 暂时在配置中修改起始地图。
 - 使用调试或“跳转”功能（如果项目提供）直接传送到特定地图或场景。

 ### 测试地图与交互

 当你主要关注某张地图或某个交互时，可以：
 - 在编辑器中打开该地图，调整布局、图块和角色摆放。
 - 运行游戏并在地图中移动，测试通行性、碰撞以及区域触发。
 - 触发事件、公共函数和粒子效果，确认视觉与玩法是否按预期工作。

 如果效果不符合预期：
 - 回到编辑器，调整地图、图块集或节点图逻辑。
 - 保存后再次运行，观察改动的影响。

 这种紧密的反馈循环正是 Ludork 工作流的核心设计之一。

 ### 使用控制台与日志

 集成的控制台与日志工具是理解运行时行为的重要助手：
 - 警告（Warning）可以提示资源缺失或路径配置不当。
 - 错误（Error）可以暴露错误数据或逻辑失败。
 - 一般信息可以帮助你追踪事件与函数的执行过程。

 在测试过程中：
 - 建议始终保持控制台可见。
 - 关注是否有重复出现的警告或错误，这通常意味着更深层次的配置问题。

 当出现问题时，可以根据控制台信息反向定位到地图、配置或公共函数中的潜在错误。

 ### 迭代建议

 为了让测试与迭代更加顺畅，建议：
 - 在运行预览前养成“随手保存”的习惯。
 - 在调试复杂行为时，尽量一次专注于一个功能或一张地图。
 - 对节点图和配置的修改尽量小步推进，这样更容易看出哪个改动产生了影响。

 随着项目推进，你可以逐渐形成自己的节奏：什么时候预览、什么时候集中调整地图或脚本、以及什么时候开始考虑打包与发布。
