 # Ludork：2D RPG 工具包 - 帮助手册
 ## 打包与发布

 本章介绍如何将 **Project（项目）** 打包成可分发的版本。Ludork 的编辑器提供了内置的打包对话框，会根据你选择的模式把运行所需的内容整理到输出目录（或生成可执行文件/应用）。

 ### 打包目标

 当你为项目打包时，一般希望：
- 生成一个独立的目录或应用，用户无需安装编辑器即可运行。
- 包含所有必要内容：`Assets/`、`Data/`、入口脚本与启动配置（例如 `Entry.py`、`Main.ini`）。
- 在需要时，将项目侧 Python 运行时代码一起纳入分发（例如 `Engine/`、`Global/`、`Source/`）。

 ### 从编辑器发起打包

 典型流程为：
 - 确保项目处于一致状态（地图、配置、公共函数、蓝图等均已保存）。
 - 打开 **打包（Pack）** 对话框，选择输出目录。
 - 选择打包模式并开始构建。

 当前 Sample 实现中仅提供 Nuitka 打包模式：
 - **Nuitka（Full）**：使用 Nuitka 将 `Entry.py` 编译为可执行文件/应用，并把 `Assets/`、`Data/`、`Main.ini` 一并带上。该模式在 Windows/macOS 会检查并要求 Python 3.12.0（用于确保构建一致性）。

 在 **打包（Pack）** 对话框中，可根据当前平台选择目标（Windows / macOS arm 等），其余打包过程对用户保持一致。

 ### 打包结果与内容

 打包完成后，你通常会得到一个输出目录或应用，其中包含：
- 项目资源：`Assets/`（图片、音频、着色器、字体等）。
- 项目数据：`Data/`（地图、图块集、配置、公共函数、本地化导出等）。
- 启动入口与启动参数：`Entry.py` 与 `Main.ini`（或其编译产物）。
- 运行时代码：随可执行文件一起被打包（由 Nuitka 通过 `--follow-imports` 追踪导入），`Engine/`、`Global/`、`Source/` 等模块都会被编译进产物。

 你可以通过以下方式测试打包结果：
- 直接运行输出目录中的可执行文件/应用。
- 检查地图、音频、字体与着色器效果是否与编辑器中的预览一致。

 如果发现缺失内容，可以检查：
- `Main.ini` 的 `script` 是否指向存在的入口脚本（默认 `Entry.py`）。
- `Assets/` 与 `Data/` 是否齐全、且资源引用路径是否仍然有效（尽量保持相对路径与目录结构不变）。

 ### 对打包结果的迭代

 打包通常并非一次性步骤。更常见的循环是：
 - 在编辑器中编辑内容，并通过内置预览反复测试。
 - 当项目达到某个稳定状态时，生成打包版本。
 - 在目标平台上测试该打包版本。
 - 返回编辑器修复问题或润色内容，然后再次打包。

 在整个过程中，保持项目结构清晰、资源分类合理，有助于打包过程顺利进行，并减少最终构建中的意外情况。
