# 23. Plugin System

Plugins extend the Ludork editor without editing the editor source tree. A plugin can add menu commands, right-click actions, right-side panels, bottom tabs, top-bar widgets, editor-localised text, and custom project data types.

Ludork ships one official reference plugin: `Plugins/OfficialLocaleTools`. It demonstrates how editor tools that are not part of the engine core can still feel native: database-menu commands, project data editing, export actions, and text-input helpers. Curve assets, by contrast, are built into the editor and runtime; see `07.Curve Assets`.

## Folder Layout

Plugins live beside `main.py`, not inside a game project.

```text
Ludork/
  Plugins/
    MyPlugin/
      Plugin.json
      Plugin.py
      locale.json
      icons/
```

`Plugin.json` and `Plugin.py` are required. `locale.json`, icons, helper modules, and other resources are optional.

The folder name must match `Plugin.json`'s `name` field exactly. Invalid plugins are skipped during editor startup and report a warning.

## Loading Model

During editor startup, Ludork scans `Plugins/*/Plugin.json`, validates each manifest, merges the optional `locale.json`, and executes `Plugin.py` with `runpy.run_path`.

`Plugin.py` receives an `EDITOR` object:

| Field | Meaning |
|---|---|
| `EDITOR.main_window` | Current main window, or `None` before the window is ready. |
| `EDITOR.project_path` | Current project root. |
| `EDITOR.plugin_dir` | Absolute path to the plugin folder. |
| `EDITOR.plugin_config` | Parsed `Plugin.json` content. |

The plugin folder is added to `sys.path` only while `Plugin.py` is being loaded, then `sys.path` is restored. Import plugin-local modules at load time, or use `EDITOR.plugin_dir` with `importlib` when delayed imports are required.

When packaging the editor, `Plugins/` is treated as a runtime asset directory and copied beside the executable output. It should not be added to Nuitka as an editor package, because plugin entry scripts are loaded dynamically through `runpy`.

## Manifest

```json
{
  "name": "MyPlugin",
  "version": "1.0.0",
  "author": "Your Name",
  "description": "Short description.",
  "minEditorVersion": "0.1.0",
  "types": ["submenu", "rightclick", "panel", "tab", "toolbar", "datatype"],
  "hooks": {
    "submenu": []
  }
}
```

Required fields are `name`, `version`, `types`, and `hooks`. Supported `types` values are:

| Type | Purpose |
|---|---|
| `submenu` | Add actions to `FILE`, `EDIT`, `GAME`, `DATABASE`, or `HELP`. |
| `rightclick` | Add actions to editor context menus. |
| `panel` | Add a right-side editor panel. |
| `tab` | Add a bottom tab. |
| `toolbar` | Add a top-bar widget. |
| `datatype` | Register a new project data folder and file extension. |
| `textInputHover` | Provide tooltips while hovering text input content. |
| `textInputHint` | Show grey non-editable hint text after text input content. |

## Menu Actions

```json
{
  "hooks": {
    "submenu": [
      {
        "parentMenu": "DATABASE",
        "label": "MY_TOOL",
        "localeKey": "MY_PLUGIN_TOOL",
        "shortcut": "Ctrl+Shift+T",
        "separatorBefore": true
      }
    ]
  }
}
```

Implement either a named hook:

```python
from PyQt5 import QtWidgets

def hook_submenu_my_tool(window):
    action = QtWidgets.QAction(ELOC("MY_PLUGIN_TOOL"), window)
    action.triggered.connect(lambda checked=False: run_tool(window))
    return action
```

or a shared fallback:

```python
def on_triggered(window, hook_type, config):
    if hook_type == "submenu" and config.get("label") == "MY_TOOL":
        run_tool(window)
```

`label` is converted to snake case for named functions, so `MY_TOOL` maps to `hook_submenu_my_tool`.

## Right-Click Actions

```json
{
  "hooks": {
    "rightclick": [
      {
        "location": "fileExplorer",
        "condition": "hit",
        "label": "INSPECT_FILE",
        "localeKey": "MY_PLUGIN_INSPECT_FILE",
        "position": "after",
        "anchor": "DELETE"
      }
    ]
  }
}
```

Named hook signature:

```python
def hook_rightclick_fileExplorer(window, condition, item_data):
    ...
```

Fallback signature:

```python
def on_triggered(window, hook_type, config, condition, item_data):
    ...
```

`condition` is usually `hit`, `empty`, or `always`. `item_data` depends on the source menu: it can be a data key, file path, actor data dictionary, layer name, table cell, or `None`.

Available locations:

| Location | Area |
|---|---|
| `mapList` | Left map list. |
| `layerTab` | Map layer tabs. |
| `editorPanel_light` | Map canvas in light mode. |
| `editorPanel_actor` | Map canvas in actor mode. |
| `blueprintGraphList` | Blueprint event graph list. |
| `fileExplorer` | Project file explorer. |
| `actorQueue` | Actor placement queue. |
| `animationAssets` | Animation asset area. |
| `commonFunction` | Common function list. |
| `tilesetList` | Tileset list. |
| `autotileList` | Autotile list. |
| `generalDataParamLabel` | General data parameter label. |
| `generalDataMember` | General data member list. |
| `generalDataTab` | General data tabs. |
| `localeSheetTab` | Official locale editor sheet tabs. |
| `localeTable` | Official locale editor table. |
| `consoleFilter` | Console filter dropdown. |

For locations with hit testing, a plugin may also declare `location_hit` or `location_empty` to target one condition directly.

## Panels, Tabs, and Toolbars

Panels are inserted into the right-side editor stack:

```json
{
  "hooks": {
    "panel": {
      "tabLabel": "MY_PANEL",
      "localeKey": "MY_PLUGIN_PANEL",
      "editModes": [0, 1, 2]
    }
  }
}
```

```python
def hook_panel(window):
    return MyPanel(window)
```

`editModes` uses `0` for tile mode, `1` for light mode, and `2` for actor mode.

Tabs are inserted into the bottom tab widget:

```python
def hook_tab(window):
    return MyBottomTab(window)
```

Toolbar entries may return any `QWidget`:

```python
def hook_toolbar_my_button(window):
    button = QtWidgets.QToolButton(window)
    button.setText(ELOC("MY_PLUGIN_BUTTON"))
    return button
```

## Text Input Hover

Plugins can supply extra tooltip text while the user hovers `QLineEdit`, `QPlainTextEdit`, or `QTextEdit` controls. The editor merges the hook result with any existing widget tooltip.

Add `textInputHover` to `types` and implement:

```python
from typing import Optional
from PyQt5 import QtWidgets

def hook_text_input_hover(
    window: QtWidgets.QMainWindow,
    widget: QtWidgets.QWidget,
    text: str,
    cursorIndex: int,
) -> Optional[str]:
    ...
```

- `text` is the widget's full current text.
- `cursorIndex` is the character index under the cursor; it may be `-1` when no position is available.
- Return `None` or an empty string to skip; non-empty results from multiple plugins are joined with blank lines in registration order.

## Text Input Hint

Plugins can show grey, non-editable hint text immediately after the editable content in `QLineEdit`, `QPlainTextEdit`, or `QTextEdit` controls.

Add `textInputHint` to `types` and implement:

```python
from typing import Optional
from PyQt5 import QtWidgets

def hook_text_input_hint(
    window: QtWidgets.QMainWindow,
    widget: QtWidgets.QWidget,
    text: str,
    cursorIndex: int,
) -> Optional[str]:
    ...
```

- `text` is the widget's full current text.
- `cursorIndex` is the current text-cursor index.
- Return `None` or an empty string to hide the hint; the first non-empty result from registered plugins is used.

`Plugins/OfficialLocaleTools` includes a built-in example: typing `{SLIME}` and other game locale keys shows the resolved text in grey after the editable content.

## Custom Data Types

A plugin can register a data type stored under `Data/{typeName}`.

```json
{
  "types": ["datatype"],
  "hooks": {
    "datatype": {
      "typeName": "Quests",
      "extension": ".quest",
      "defaultFormat": "json"
    }
  }
}
```

```python
def hook_datatype():
    return {
        "loader": load_quest_file,
        "editor": create_quest_editor
    }
```

If no loader is provided, `defaultFormat: "json"` uses JSON loading and `defaultFormat: "dat"` uses Ludork's binary data loader.

Loaded plugin data participates in editor data snapshots, modified checks, rename/delete tracking, and save/reference bookkeeping. Re-initialising game data clears plugin data first, so stale or duplicated entries should not remain after a project reload.

## Plugin Localisation

Add `locale.json` beside `Plugin.py`. The language-first form is preferred:

```json
{
  "en_GB": {
    "MY_PLUGIN_TOOL": "My Tool"
  },
  "zh_CN": {
    "MY_PLUGIN_TOOL": "我的工具"
  }
}
```

Plugin locale keys are merged into the editor locale table before hooks are installed, so `ELOC("MY_PLUGIN_TOOL")` can be used inside plugin code and manifests.

## Official Example: Locale Tools

`Plugins/OfficialLocaleTools` is the recommended starting point when building editor extensions. It is shipped with Ludork but implemented as a normal plugin, not as hard-coded editor logic.

**What it adds**

- `DATABASE` menu entries for opening the locale table and exporting language files.
- Inline grey hint preview for `{LOCALE_KEY}` placeholders in text inputs.
- Plugin-local UI in `OfficialLocaleToolsLocaleEditor.py`.
- Plugin-local Excel import/export in `OfficialLocaleToolsLocaleIO.py`.
- `locale.json` entries for its menu and dialog text.

**Typical workflow**

1. Open the locale table from the database menu.
2. Edit keys in the plugin editor or in `Data/Locale/Locale.xlsx`.
3. Run the export command so `Data/Locale` files are refreshed.
4. Save the project and test text in `Test Game`.

**What to copy**

- `Plugin.json` manifest layout and `submenu` hook declarations.
- Named hook functions such as `hook_submenu_view_locale_table`.
- Optional `textInputHint` integration for data-aware inline previews.
- Separation between plugin UI (`OfficialLocaleToolsLocaleEditor.py`) and IO helpers (`OfficialLocaleToolsLocaleIO.py`).

Use this plugin as the practical template when moving editor tools out of the core UI. For the author-facing localisation checklist, see `06.Data Localisation and Testing`.
