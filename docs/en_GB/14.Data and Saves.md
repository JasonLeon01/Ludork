# 14. Data, Localisation, and Saves

This chapter covers script access to project data, game localisation, and save files.

## Data Locations

Projects store runtime data in `Data/`:

- `Data/Animations`
- `Data/AutoTiles`
- `Data/CommonFunctions`
- `Data/Configs`
- `Data/General`
- `Data/Locale`
- `Data/Maps`
- `Data/Tilesets`

Scripts normally access these through `Source.Data` or config helpers rather than opening files directly.

## Loading Project Data

`Source.Data` owns the template project's runtime data registry.

Load functions:

- `loadAnimations()`
- `loadCommonFunctions()`
- `loadTilesets()`
- `loadAutoTiles()`
- `loadGeneralData()`

Get functions:

- `getAnimation(name)`
- `getTileset(name)`
- `getAutoTile(name)`
- `hasAutoTile(name)`
- `getGeneralData(name)`
- `getGeneralClassData(key)`
- `getGeneralEnemyData(key)`
- `getGeneralSpecialData(key)`
- `getGeneralStateData(key)`
- `getGeneralItemData(key)`
- `getGeneralEquipData(key)`
- `getClass(classPath)`
- `getClassData(classPath)`
- `getCommonFunction(name)`
- `genGraphFromData(data, parent=None, parentClass=None)`
- `genActorFromData(actorData, layerName)`

Example:

```python
from Source import Data

Data.loadTilesets()
tileset = Data.getTileset("Tileset_01")
```

The initial scene is the usual place to load project data.

## General Data

`Data/General` contains RPG records such as items, equipment, enemies, states, specials, and classes.

Use:

```python
itemData = Data.getGeneralData("Item")
```

Project code in `Source.Item`, `Source.Equip`, `Source.Enemy`, `Source.Battler`, and `Source.Infos` builds typed gameplay objects from this data. `Special.dat` stores enemy specials, such as `Poisoning` in the current template; `State.dat` builds `StateInfo` objects that can be attached to players or enemies.

## Config Data

`Source.System.init()` reads:

- `Data/Configs/System.json`
- `Data/Configs/Audio.json`

Runtime getters expose common values:

- `getTitle()`
- `getFonts()`
- `getFontSize()`
- `getWindowskinName()`
- `getStartMap()`
- `getStartPos()`
- sound getters such as `getDecisionSE()`
- `getTitleBGM()`

Example:

```python
from Source import System as GameSystem

startMap = GameSystem.getStartMap()
decisionSE = GameSystem.getDecisionSE()
```

## Locale Data

`Entry.py` initialises locale data:

```python
Engine.Locale.init("./Data/Locale")
```

Use:

```python
text = LOC("TITLE_START")
```

`Global.System.setLanguage(language)` applies and saves the current language. `saveLanguage(language)` saves without applying immediately.

## Editing Localisation

Projects keep the editable localisation source at:

```text
Data/Locale/Locale.xlsx
```

The exported runtime files are stored beside it:

```text
Data/Locale/en_GB
Data/Locale/zh_CN
```

The workbook format is:

- The first row is the header row.
- The first column must be `ID`.
- Each following column is a language key, such as `zh_CN` or `en_GB`.
- Each row after the header defines one text ID.
- Different sheets may be used to organise text, but IDs should remain unique and stable.

Example:

| ID | zh_CN | en_GB |
| --- | --- | --- |
| TITLE_START | 开始游戏 | Start |
| TITLE_EXIT | 退出 | Exit |

After editing `Locale.xlsx`, export it so the runtime language files are regenerated. If you edit through the project's localisation editor, saving the workbook also exports the runtime files. If you edit the workbook externally, run the localisation export command before testing or packaging.

Keep placeholders identical between languages:

| ID | zh_CN | en_GB |
| --- | --- | --- |
| GET_ITEM | 获得了 {name} | Got {name} |

Do not rename `{name}` differently in another language. Runtime formatting expects the same placeholder names.

## Save Files

The template save flow is managed by `Source.Save`, `Source.GameInstance`, and `WindowSaveLoad`.

`GameInstance` stores:

- the current region and player list.
- game variables.
- cached map file.
- destroyed actor records.
- persistent terrain edits.
- recorded floor teleporter points.
- save thumbnail screenshot data.
- player inventory, equipment, attributes, and state IDs.

The map scene provides the save source:

```python
def _getSaveSource(self) -> GameInstance:
    return self.inst
```

After loading, the map scene calls `applyLoadedGame(inst)` to rebind the player to UI, reload the cached map, and place the player at the saved position.

## Destroyed Actors

To persist removed map actors:

```python
scene.recordDestroyedActor(actor)
actor.destroy()
```

When a map is loaded, the scene checks the saved destroyed actor tags and destroys matching actors again.

## Terrain Edits

To persist map terrain changes, use the terrain destruction helpers on the map or node functions:

```python
gameMap.destroyTerrain("Ground", (6, 12), None)
gameMap.destroyTerrainList("Ground", [(6, 12), (7, 12)], None)
```

These calls edit the current map and record the change through `GameInstance.recordTerrainDestruction()`. When loading old saves, `GameInstance` accepts both dictionary-style and list-style terrain records.

## Floor Teleporter Points

`GameInstance.recordTelepoint(mapPath, telepoint)` stores a teleport position on a map. `WindowFloorTeleporter` reads recorded points in the current region, shows the map list and preview, then asks the map scene to switch to the selected map and position when confirmed.

## Screen Snapshots

The map scene captures the current canvas before opening the menu so save files can use a screen image.

Use:

- `Source.System.getSavedScreenImage()`
- `Source.System.setSavedScreenImage(image)`

## Direct File Helpers

For custom files:

```python
from Engine.Utils import File

data = File.getJSONData("./Data/Configs/System.json")
obj = File.loadData("./Data/Maps/Map_01.dat")
File.saveData("./Save/custom.dat", obj)
```

Keep save data compatible when changing its structure. Prefer adding optional fields over changing existing meanings.
