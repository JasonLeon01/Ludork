# 12. Input and Controls

The Sample runtime normalises keyboard, mouse, touch, and gamepad input through the shared input layer.

## Default Controls

**Keyboard**

Arrow keys move the player or menu selection. `Enter` and `Space` confirm. `Escape` cancels, closes a window, or opens the in-game menu.

**Mouse**

Left click attempts map pathing or confirms a UI choice. Right click cancels, closes a window, or opens the in-game menu.

**Touch**

Tap attempts map pathing. Two-finger touch acts like right-click cancel.

**Gamepad**

Directional axes navigate or move. The confirm and cancel buttons follow the runtime input mapping.

## Action Helpers

Game code should use action helpers instead of checking only one physical key.

```python
from Engine import Input

if Input.isActionTriggered(Input.getConfirmKeys(), handled=True):
    confirm()

if Input.isActionTriggered(Input.getCancelKeys(), handled=True):
    cancel()
```

## Map Click Movement

Left mouse and touch input can request automatic pathing. The map scene consumes this input when a blocking window is open so a click used for UI does not also move the player.
Automatic pathing avoids non-colliding actors with implemented `onOverlap` events; manual movement can still step onto them to trigger the interaction.

## Input Mode

The input layer tracks whether the active device is mouse or gamepad. UI controls can use this to decide whether to show mouse hover behaviour or gamepad selection behaviour.

## Custom Input Advice

When adding new gameplay input, route it through `Engine.Input` and mark it handled when consumed. This keeps menus, windows, click movement, and touch cancel from fighting each other.
