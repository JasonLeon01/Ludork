# 12. Windows, Menus, and Tips

The Sample game provides a set of windows used by title flow, map flow, events, and saves.

## Title Windows

**Command window**

Shows `Start`, `Continue`, `Config`, and `Exit`.

**Save-load window**

Used by `Continue` to load existing save slots.

**Config window**

Changes runtime settings and writes them back through the config system.

## Map Windows

**HUD**

Shows key player values and should update after rewards, equipment changes, item use, damage, healing, and state changes.

**Menu**

Opened by cancel input on the map. It gives access to items, equipment, save, and return-to-title commands.

**Item window**

Shows usable items and handles item use.

**Equipment windows**

Show equipment slots and available equipment.

**Save window**

Writes current progress into the save folder.

## Focus Groups and Window Navigation

The Sample runtime separates `active` from keyboard focus. `active=True` means the window or focus group can become the target of keyboard or gamepad input; `active=False` means it is currently not selectable. Focus itself is global and unique, managed by the scene `FocusManager`.

**Focus groups**

Related windows are placed in `FocusGroup`s. Direction keys first move the cursor inside the current window. When the cursor reaches a boundary and a directional neighbour exists, focus can move to that neighbouring group. If an intermediate group cannot be entered, for example because it is `active=False`, navigation keeps looking in the same direction for the next enterable group.

**Confirm-driven child windows**

Equipment, save/load, shop, and floor teleporter windows do not enter child lists through direction keys alone. They explicitly move focus after confirming the current command, and cancel returns to the parent layer:

- Confirming Equipment focuses the equipment slot list; confirming a slot focuses the available-equipment list; cancelling in that list returns to the slot list.
- Confirming `Load` or `Save` focuses the save slot list; cancelling in the slot list returns to the `Load / Save` command bar.
- Shop and floor teleporter windows follow the same "confirm to enter, cancel to return" rule.

**Mouse and focus**

Visible windows can still receive mouse hover. Hovering a focusable selectable window can sync the keyboard cursor to that window. Clicks, wheel input, and touch confirmation still require `active=True`, so inactive layers are not confirmed accidentally.

## Event Windows

**Message window**

Displays dialogue and selections. It can position itself relative to a map actor when events provide a reference.

**Shop window**

Handles buying and selling for a supplied item list.

**Enemy book**

Shows enemies on the current map and previews battle damage.

**Floor teleporter**

Shows recorded floors and map previews.

## Window Rule

Blocking windows should consume cancel and click input, stop map click movement, and restore player movement after closing.

Custom windows that should participate in cross-window navigation should register `FocusGroup`s in the scene, define directional neighbours, and request focus when confirm, cancel, or business state changes move the user to another layer. Do not rely on Z-order to decide which window receives keyboard input.
