# 05. Blueprints and Actors

This chapter explains how to maintain blueprints, actor instances, and blueprint events in the editor.

## Blueprint Categories

Blueprints are stored in `Data/Blueprints`. The template project contains common categories:

- `Actors`
- `Consumables`
- `Doors`
- `Enemies`
- `Equips`
- `Items`
- `Teleportations`

Categories are only file organisation. Behaviour is controlled by a blueprint's parent, properties, components, and event graph.

## Creating a Blueprint

When creating a blueprint, choose its parent first. The parent can be a project class or an existing project blueprint.

Common parents include player, enemy, item, equipment, door, teleporter, and plain actor classes. When inheriting from an existing blueprint, the new blueprint only needs to override values that differ from the parent.

After creating the blueprint, save the project so the blueprint file is written to `Data/Blueprints`.

## Blueprint Properties

Use the blueprint property panel to set texture, default rectangle, collision, ticking, material, shader, speed, and related attributes.

Common properties:

- `texturePath`: actor or object texture, usually relative to `Assets`.
- `defaultRect`: rectangle selected from the texture.
- `collisionEnabled`: whether the actor blocks movement.
- `tickable`: whether the actor receives frame events.
- `speed`: movement speed.
- `material`: rendering material.
- `shaderPath`: shader file to use.

Only values that differ from the parent need to be saved; inherited values continue to come from the parent.

## Components

Components in a blueprint should be maintained as grouped data, not flattened into ordinary fields.

Common components:

- `infoComp`: battle or identity attributes such as `HP`, `MAXHP`, `ATK`, `DEF`, `EXP`, `GOLD`, `LEVEL`, and `CLASS`.
- `lightComp`: self-light settings. Enable `bSelfLight`, then set `lightColor` and `lightRadius`.
- Child actor component: lets one actor spawn and carry another actor.

Component fields use type-aware controls. Booleans use checkboxes, numbers use numeric inputs, and colour or list fields are edited by element.

## Event Graphs

Event pages edit blueprint event graphs. Common events include:

- `onCreate`
- `onTick`
- `onLateTick`
- `onFixedTick`
- `onCollision`
- `onOverlap`
- `onDestroy`

`onCollision` is useful for logic triggered when movement is blocked, such as touching a door or dialogue object. `onOverlap` is useful for logic after entering the same cell, such as picking up an item.

The template `BP_Actor_TestDialogue` demonstrates a collision flow that shows dialogue, shows a selection, flashes the screen, shakes the screen, opens a shop, and starts weather.

## Placing Actors on Maps

In `Actor Mode`, place blueprint instances on a map. After placing one, check:

- Whether the position is the expected cell.
- Whether the tag is stable and readable.
- Whether instance properties only override what is special for this map.
- Whether texture rectangle, collision, and light effects are correct.

If multiple map instances should share the same behaviour, edit the blueprint. If only one placement is special, edit the instance properties.
