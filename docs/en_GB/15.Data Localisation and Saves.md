# 15. Data, Localisation, and Saves

This page explains how Sample game data, localisation, and saves connect at runtime.

## Data References

Maps, blueprints, general data, localisation, and assets reference one another by names, tags, keys, and paths. Changing a name is more than a cosmetic edit: every referencing map, event, window, and save path must still resolve.

## Localised Text

The runtime locale module loads exported language files from `Data/Locale`. Editing and exporting those files is handled by the official locale tools plugin (`Plugins/OfficialLocaleTools`), not by built-in engine UI. See `06.Data Localisation and Testing` for the editor workflow and `23.Plugin System` for the plugin layout.

```python
from Engine import Locale

title = Locale.GetContent("GAME_TITLE")
languages = Locale.GetLocaleKeys()
```

Text wrapped in braces, such as `{SLIME}`, is resolved by runtime localisation helpers when data is loaded or displayed.

## Placeholder Rule

All languages must keep the same placeholder names. If `en_GB` contains `Found {count} keys`, then `zh_CN` must also contain `{count}`.

## Save Content

The Sample save system records:

- player position, values, inventory, equipment, and states;
- current map and scene-relevant state;
- obtained item/equip IDs used to show pickup detail messages only the first time;
- added actors, moved actors, and removed actors such as opened doors and collected pickups;
- changed terrain;
- recorded floor teleport points;
- a screenshot and slot detail data.

## Save Functions

Visual events can call save helpers when a custom flow needs direct control:

```python
from Source.NodeFunctions.Save import GetSavePath, SaveGame, LoadGame

path = GetSavePath(1, "dat")
SaveGame(path)
result = LoadGame(path)
```

## Compatibility Advice

When adding new saved fields, keep old saves readable. Missing fields should have defaults, and removed actors or changed terrain should continue to be identified by stable map names and tags.
