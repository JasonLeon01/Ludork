# 15. Node Functions and Events

Ludork can execute node graph data at runtime. The template project uses node functions for event-like gameplay logic and common functions.

This chapter describes the script side only.

## Common Functions

`Source.Data.getCommonFunction(name)` loads a graph from `Data/CommonFunctions`.

It looks for:

- `Data/CommonFunctions/<name>.dat`
- `Data/CommonFunctions/<name>.json`

Example:

```python
graph = Data.getCommonFunction("PrintTest")
```

`Data.genGraphFromData(data, parent=None, parentClass=None)` builds a graph object from loaded data.

## Actor Graphs

Actors generated from map data receive a graph:

```python
actor.setGraph(Data.genGraphFromData(classData["graph"], actor, classModel))
```

Actor events then dispatch into this graph through registered event names.

## RegisterEvent

Use `@RegisterEvent` to mark an actor method as a blueprint event.

```python
from Engine.Gameplay.Actors import Actor

class Door(Actor):
    @RegisterEvent
    def onOverlap(self, other: list[Actor]) -> None:
        pass
```

Built-in actor events include:

- `onCreate`
- `onTick`
- `onLateTick`
- `onFixedTick`
- `onDestroy`
- `onCollision`
- `onOverlap`

State data in `Source.Infos.StateInfo` also uses registered events. The current events are:

- `onWalk`
- `onHookTriggered`

## ExecSplit

`@ExecSplit(...)` describes execution branches for node logic.

Example from actor movement:

```python
@ExecSplit(success=(True,), fail=(False,))
def MapMove(self, offset) -> bool:
    ...
```

The returned value chooses a branch.

## Latent

`@Latent(...)` marks a function that returns a condition callable. The graph can wait until the condition returns a completion value.

Example shape:

```python
@Latent(TimeUp=(True,))
def waitSomething(self) -> Callable[[], bool]:
    def condition() -> bool:
        return ready
    return condition
```

Scene timers, dialogue, and selections use this style.

## ReturnType

`@ReturnType(name=Type)` records named return values for tooling and node execution.

```python
@ReturnType(actors=list[Actor])
def getAllActors(self) -> list[Actor]:
    ...
```

## TypeAdapter

`@TypeAdapter` adapts node and script input values before the function body runs.

Example:

```python
@TypeAdapter(position=(tuple, Vector2f))
def setLightPosition(self, light, position):
    ...
```

This allows node data or script calls to pass simple tuples.

## Source.NodeFunctions

Project node functions live in `Source/NodeFunctions`.

The template project includes groups such as:

- `Math`
- `String`
- `Container`
- `Utils`
- `System`
- `Scene`
- `Player`
- `State`
- `Save`
- `Movement`
- `Mota`

Common responsibilities in the current template:

- `System`: sound, music, video, transitions, flash, shake, weather, and filters.
- `Scene`: map changes, Game Over, telepoint recording, terrain reads/writes, persistent terrain edits, and shop opening.
- `Player`: items, equipment, attributes, HP, gold, EXP, and player lookup.
- `State`: state host access, event arguments, arbitrary battler attributes, and cross-object state add/remove.
- `Movement`: actor route and pathfinding movement.
- `Mota`: enemy book, floor teleporter, and current region.

Use these files as the pattern for new node-callable gameplay functions.

## Event Safety

Node functions may be called from graph execution rather than a direct scene method. Keep them small and explicit:

- Validate important inputs.
- Prefer runtime APIs over direct private state changes.
- Use `System.setScene()` for scene changes.
- Use `SceneMap.showMessage()` and `showSelection()` for blocking dialogue.
- Use `GameMap` methods for actor lookup and spawning.

## Example: A Node-Friendly Function

```python
from Engine import ReturnType
from Global import System
from Source.Scenes.SceneMap import Scene as SceneMap

@ReturnType(scene=SceneMap)
def getMapScene() -> SceneMap:
    scene = System.getScene()
    assert isinstance(scene, SceneMap)
    return scene
```

Keep return names meaningful. They appear in node and documentation metadata.
