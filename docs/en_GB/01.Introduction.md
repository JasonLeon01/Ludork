# 01. Introduction

Ludork is a 2D RPG toolchain made of an editor, a default Sample game, and a small runtime framework. This manual follows the tone and layout of a classic help file: each page begins with the purpose of the feature, then lists the relevant commands, data, properties, and checks.

## Manual Sections

**Editor Manual**

Chapters `02` to `07` describe how to use the editor. These chapters avoid runtime code and focus on the visible workflow: opening a project, editing data, choosing assets, saving, and testing.

**Sample Game Reference**

Chapters `08` to `18` describe the default Sample game. These chapters may mention data files, runtime behaviour, node functions, and small code examples where that helps explain how the game side works.

**Blueprint Scripting Reference**

Chapter `19` documents decorators such as `@Meta`, `@ExecSplit`, `@Latent`, and `@LoopNode`, plus parameter-default conventions. Chapter `20` documents General Data editing. Chapter `21` covers practical tips for authoring default Sample blueprints. Chapter `22` lists blueprint-visible API signatures.

**Editor Extensions**

Chapter `23` documents the plugin system, including the official locale tools plugin as a reference implementation.

## Ludork Default Project

The default project is a Mota-style RPG template. It includes maps, autotiles, a player blueprint, enemies, doors, keys, consumables, equipment, an enemy book, floor teleporting, shops, saves, settings, weather, fog, sound effects, and title music.

## Default Values

**Project marker**

`Main.proj` marks a folder as a Ludork project.

**Startup script**

`Main.ini` points to `Entry.py` in the Sample project.

**Languages**

The Sample project includes `zh_CN` and `en_GB`. The English documentation and language key are `en_GB`.

**Display**

The default content size is `640 x 480`, the tile size is `32`, and the default scale is `2.0`.

**New game position**

New games start on `Map_01.dat` at tile position `(6, 12)`.

## How to Read This Manual

If you are making content, start with the editor chapters and follow the save-and-test rhythm. If you are changing the Sample game behaviour, use the later reference chapters to find the relevant data or runtime hook.

When a page says to verify a change, it means testing the actual project through `Test Game`, not only checking that the editor field was changed.
