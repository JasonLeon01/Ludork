# 01. Introduction

This manual describes the Ludork game runtime, project scripts, and common editor workflows for a Ludork project. It is written for project authors who want to modify game logic, scenes, actors, windows, data loading, audio, and effects.

When a subject can be handled either by data or by script, the scripting chapters explain the script side, while the editor workflow chapter explains which data should be maintained in the editor.

## How to Read This Manual

The style follows a reference-manual layout:

- Concepts explain where a feature belongs.
- Class pages list important methods and their purpose.
- Examples show the normal script shape used by the template project.
- Notes describe conventions that matter at runtime.

Most examples assume the working directory is the project root.

## Runtime Overview

A Ludork game project is a Python package-like folder. A typical project is arranged as follows:

```text
Project/
  Entry.py
  Main.ini
  Assets/
  Data/
  Engine/
  Global/
  Source/
  Save/
  Temp/
```

`Entry.py` is the process entry point. It loads locale data, initialises the node graph latent system, initialises `Global.System`, creates the first scene, and then runs the active scene until the game exits.

`Engine` contains low-level types, input, UI controls, decorators, file helpers, node graph execution, gameplay primitives, and SFML/PySF exports.

`Global` contains shared game systems such as the window, scene stack, camera, map, weather, fog, resource managers, video playback, and scene base class.

`Source` contains the project's game rules. In the template project it includes the title scene, map scene, player, battlers, inventory, equipment, save files, windows, data access, and node functions.

## Main Packages

### Engine

Import `Engine` for core objects and helpers:

```python
import Engine
from Engine import Vector2u, Vector2f, Color, Direction
from Engine import Input
```

Important global values:

- `Engine.CellSize`: tile size in pixels. The template project uses `32`.
- `Engine.GameSize`: logical game resolution.
- `Engine.Scale`: display scale factor.
- `Engine.GameRunning`: main loop flag.

### Global

Import `Global` for game-wide runtime systems:

```python
from Global import System, SceneBase, GameMap, Camera
from Global import Manager
```

Use `Global.System` for window state, scene switching, transitions, screen effects, language, frame rate, and audio settings.

### Source

Import `Source` for project-specific game rules:

```python
from Source import Data
from Source.GameInstance import GameInstance
from Source.Scenes.SceneMap import Scene as SceneMap
```

Scripts in `Source` are expected to change from game to game. Scripts in `Engine` and `Global` are runtime foundations.

## Script Conventions

Ludork scripts use Python 3.12 syntax and type annotations. The project commonly uses lower camel case method names, matching the existing runtime code.

Many runtime methods accept Python tuples or lists for vector-like values because decorators adapt them to SFML vector types:

```python
actor.setMapPosition((3, 5))
plainText.setOrigin((16, 8))
```

After adaptation, treat the stored value as an engine object such as `Vector2u`, `Vector2f`, or `IntRect`, not as the original tuple.

## Running a Project

Run a project from its root:

```bash
python Entry.py
```

The template entry point also starts `debugpy` on `localhost:2333` when `debugpy` is installed. If it is missing, the game continues normally.
