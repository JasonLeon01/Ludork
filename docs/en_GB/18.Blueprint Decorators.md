# 18. Blueprint Decorators

The Sample runtime provides blueprint decorators in `Sample/Engine/Decorators.py`. The editor reads them when loading Python scripts to control node presentation, parameter editors, execution flow, and return pins.

Decorators live in the `Engine` package and are injected as global names by `Engine/__init__.py`, so scripts under `Source/`, `Global/`, and similar paths can write `@Meta(...)` without an explicit import.

## Decorator Overview

| Decorator | Applies to | Purpose |
| --- | --- | --- |
| `@Meta(...)` | Functions, methods, classes | Editor metadata: display names, descriptions, specialised parameter editors |
| `@ExecSplit(...)` | Functions, methods | Declares execution outputs such as `default` or `Success` |
| `@Latent(...)` | Functions, methods | Declares latent/async outputs that wait for a condition |
| `@LoopNode(...)` | Functions, methods | Marks a synchronous loop controller such as `ForLoop` or `ForEach` |
| `@ReturnType(...)` | Functions, methods | Declares data output pin names and types |
| `@TypeAdapter(...)` | Functions, methods | Converts parameter types before the call |
| `@RegisterEvent` | Functions, methods | Registers a function as a blueprint event-page entry |
| `@InvalidVars(...)` | Classes | Hides class variables from the blueprint attribute editor |
| `@RectRangeVars(...)` | Classes | Edits rectangle fields with an image range selector |

Functions with `@ExecSplit`, `@Latent`, `@LoopNode`, or `@ReturnType` also receive `_refLocal` and therefore appear in the blueprint node picker. Functions with only `@Meta` are not exposed automatically.

## `@Meta` Editor Metadata

`@Meta(...)` attaches a `_meta` dictionary to a function, method, or class. The editor currently recognises these keys:

- `DisplayName` and `DisplayDesc`: localised node names and descriptions, usually written as `LOC("KEY")`. The editor evaluates these expressions when building node titles and the function-picker search cache.
- `VariableDisplayNames` and `VariableDisplayDescs`: class-variable display labels and tooltips, written as dictionaries such as `{"speed": 'LOC("ACTOR_VAR_SPEED")'}`. The dictionary keys are variable names; the string values are evaluated the same way as `DisplayName` / `DisplayDesc`.
- `DropBox`: combo-box options for node parameters, for example `{"attr": ["pitch", "volume"]}`.
- `Rely`: edit dependencies between fields or parameters, for example `{"target": ["source", True]}`, or the dict form `{"target": {"source": "source", "op": "!=", "value": None}}` (`key` / `var` are accepted instead of `source`). `Scene.ShowVoiceMessage` uses this to expose `minDistance` only when `refActor` is not `None` (formerly `ShowVoiceRefMessage`).
- `PathVars`: path fields or node parameters rooted under `Assets`, for example `[("texturePath", "Characters")]`. String-only items such as `["texturePath"]` default to the `Characters` assets directory.
- `ColourVars`: colour fields or parameters, for example `["lightColour"]`. Values store RGBA tuples but display through a colour swatch editor.
- `ProgressVars` / `SliderVars` / `RangeVars`: numeric fields or parameters displayed as a draggable slider plus value box. Use a dictionary to define the range, for example `{"hue": (0.0, 360.0, 1.0)}` where the tuple is `(min, max, step)`.
- `Vector2Vars`, `Vector2fVars`, `Vector2iVars`, `Vector2uVars`, `PairVars`, `PairFloatVars`, `PairIntVars`: two-component fields or node parameters displayed as x/y numeric editors.
- `Vector3Vars`, `Vector3fVars`, `Vector3iVars`, `Vector3uVars`: three-component fields or node parameters displayed as x/y/z numeric editors.
- `MoveRouteVars`: movement-route node parameters displayed with a map reference route editor. Values are saved as relative `(dx, dy)` steps.
- `Transfer`: map coordinate node parameters displayed with a map tile picker dialog. Format: `[("posVarName", "mapVarName")]`. Each tuple pairs the coordinate parameter with another parameter that supplies the map key used to pre-select the map. If that map parameter is empty or refers to a non-existent map when the dialog is confirmed, it is automatically filled with the selected map key.
- `BlueprintClassVars`: blueprint class path node parameters, for example `["bpPath"]`. The editor shows the same project class/blueprint selector used when creating a blueprint, while saved values remain class path strings.
- `CommonFunctionVars`: common function name node parameters, for example `["commonFunctionName"]`. The editor shows a selector backed by the current project's common functions, while saved values remain plain strings.
- `GeneralDataVars`: General Data member or animation-key fields and node parameters, for example `[("itemID", "Item"), ("animKey", "ANIMATION")]`. The editor shows a combo box, while saved and runtime values remain plain strings.
- `InvalidVars`: class variables hidden from the blueprint attribute editor, for example `["defaultRect"]`. Equivalent to the class decorator `@InvalidVars(...)`.
- `RectRangeVars`: rectangle fields edited with an image range selector, for example `{"defaultRect": "texturePath"}`. Equivalent to the class decorator `@RectRangeVars(...)`.

Example:

```python
@Meta(
    DisplayName='LOC("PLAY_SOUND")',
    DisplayDesc='LOC("PLAY_SOUND_DESC")',
    PathVars=[("soundFileName", "Sounds")],
)
@ExecSplit(default=(None,))
def PlaySound(soundFileName: str, applyFilter: bool = False) -> None:
    ...
```

## `GeneralDataVars` References

Use `GeneralDataVars` on node parameters when a string should reference a General Data member ID:

```python
@Meta(
    DisplayName='LOC("USE_ITEM")',
    GeneralDataVars=[("itemID", "Item"), ("animKey", "ANIMATION")],
)
@ExecSplit(default=(None,))
def UseItem(itemID: str = "", animKey: str = "") -> None:
    ...
```

The same metadata works for blueprint class variables:

```python
class Chest:
    itemID: str = ""
    openAnim: str = ""

    _meta = {
        "VariableDisplayNames": {
            "itemID": 'LOC("CHEST_ITEM_ID")',
            "openAnim": 'LOC("CHEST_OPEN_ANIM")',
        },
        "VariableDisplayDescs": {
            "itemID": 'LOC("CHEST_ITEM_ID_DESC")',
            "openAnim": 'LOC("CHEST_OPEN_ANIM_DESC")',
        },
        "GeneralDataVars": [
            ("itemID", "Item"),
            ("openAnim", "ANIMATION"),
        ]
    }
```

The second tuple item is a key from `GameData.generalData` (`Class`, `Enemy`, `Equip`, `Item`, `Player`, `Special`, or `State`). The special uppercase value `ANIMATION` reads animation keys instead. The first option is `— None —`, which saves an empty string. Missing or empty data sources are handled without an error. `GeneralDataVars` takes precedence over `DropBox`, while specialised metadata such as `PathVars`, `ColourVars`, and vector types takes precedence over `GeneralDataVars`.

## `@ExecSplit` Execution Outputs

`@ExecSplit` declares execution pins that fire after the function completes. Keys are output labels; values are tuples reserved for runtime extensions, usually `(None,)` or branch indices.

Common patterns:

```python
@ExecSplit(default=(None,))
def DictAdd(dict_: Dict, key: Any, value: Any) -> None:
    ...

@ExecSplit(Success=(0,), Failed=(1,))
def RemoveItem(itemID: str, count: int = 1) -> int:
    ...

@ExecSplit(TRUE=(0,), FALSE=(1,))
def IF(condition: bool = False) -> int:
    ...
```

## `@Latent` Latent Outputs

`@Latent` marks nodes that must wait for a condition before continuing. The function usually returns a callable; the blueprint waits on the matching output until that callable returns `True`.

```python
@Latent(Finished=(True,))
def RequestTransition(transitionName: str = "", transitionTime: float = 1.0) -> Callable[[], bool]:
    ...

@Latent(Closed=(True,))
def OpenShop(items: List[str], canSell: bool = True) -> Callable[[], bool]:
    ...
```

## `@LoopNode` Loop Controllers

`@LoopNode(kind)` marks a function as a synchronous loop controller. It is used together with `@ExecSplit` and `@ReturnType`: execution outputs define `LoopBody` and `Completed`, while return pins provide the per-iteration values consumed by the loop body.

```python
@LoopNode("ForLoop")
@ExecSplit(LoopBody=("__loop_body__",), Completed=("__loop_completed__",))
@ReturnType(index=int)
def ForLoop(firstIndex: int = 0, lastIndex: int = 0, step: int = 1) -> tuple[int, int, int]:
    ...
```

Loop controllers are currently synchronous. A `LoopBody` branch must not contain latent nodes; the runtime raises an error if a latent node is reachable from the loop body.

## `@ReturnType` Data Outputs

`@ReturnType` declares data pins for pure-data nodes. Keys are pin names; values are type annotations used for documentation and editor hints.

```python
@ReturnType(value=bool)
def DictContains(dict_: Dict, key: Any) -> bool:
    ...

@ReturnType(index=int)
def ListFind(list_: List[Any], value: Any) -> int:
    ...
```

## `@TypeAdapter` Parameter Adaptation

`@TypeAdapter` converts parameters before the function runs. Each parameter maps to a tuple:

- `(originType, targetType)`: uses `targetType` as the converter;
- `(originType, targetType, converter)`: uses the specified `converter` callable.

`originType` may be a single type or a list of types. When the incoming value is a `list` or `tuple`, the converter is called with unpacked values.

```python
@TypeAdapter(position=((list, tuple), Vector2i, Vector2i))
def setMapPosition(self, position: Vector2i) -> None:
    ...
```

## `@RegisterEvent` Event Registration

`@RegisterEvent` marks a function as a blueprint event-page entry and automatically adds a `default` execution output. It may be used with or without parentheses.

```python
@RegisterEvent
def onCreate(self) -> None:
    ...
```

## Class Decorators

`@InvalidVars("fieldA", "fieldB")` and `@RectRangeVars(defaultRect="texturePath")` may also be applied directly to a class. They behave the same as `@Meta(InvalidVars=...)` and `@Meta(RectRangeVars=...)`.

## Parameter Defaults

Node parameters with Python signature defaults are still shown in the blueprint editor. Unconnected pins use the default from source code.

Defaults should be Python basic values (`None`, `bool`, `int`, `float`, `str`, or containers made from those values). Defaults that refer to engine or editor objects should be written as strings and handled explicitly inside the function.

List and dict node parameters that represent "empty" or "clear" should default to `[]` or `{}` rather than `Optional[...] = None`, so blueprint pins stay typed as plain `List[...]` / `Dict[...]`. Parameters that are genuinely absent at runtime may still use `Optional[...] = None` (for example `TriggerEventBus.kwargs`).

Many `Source.NodeFunctions` entries now ship with practical defaults, for example:

- `String` helpers default string parameters to `""` or `"0"`;
- `Math` helpers default vector components to `0` and comparison operands to `0` / `1`; `Max` / `Min` default `values` to `[]`;
- `Movement.SetMoveRoute` and `Actor.setRoute` default `route` to `[]` (empty list clears the route);
- `Player` helpers default quantity parameters to `1`;
- `System` helpers default `applyFilter` to `False` for `PlaySound` / `PlayMusic` and colour channels to `255` for `FlashScreen`; `Scene.ShowVoiceMessageByTag` is the tag-based voice node and `Scene.ShowVoiceMessage` defaults `minDistance` to `64.0`.

See `21.API Reference/02.NodeFunctions/` for the full signatures.

## Stacking Order

Recommended order from outer to inner: `@Meta` → `@LoopNode` when needed → `@ExecSplit` / `@Latent` / `@ReturnType` → `@TypeAdapter` when needed. `@Meta` affects editor presentation only; execution and return decorators define the pin layout in the blueprint graph.
