 # Ludork: 2D RPG Toolkit - Help Manual
 ## Advanced Topics
 
 This chapter collects advanced workflows for extending Ludork, performance-oriented practices, and troubleshooting guidance. It is primarily intended for developers who customize the editor, extend the runtime, or package builds for distribution.
 
 ### Extending the Editor
 
 #### Editor extensions (EditorExtensions/)
 
 Ludork can load native editor-side extensions from the `EditorExtensions/` directory. These are typically compiled Python extension modules (for example, `.so` / `.pyd`) that accelerate editor workloads.
 
 A representative example is tilemap rendering in the map editor panel, which tries to import a native renderer and falls back to a pure-Python renderer if the extension is unavailable.
 
 Practical notes:
 - Native editor extensions are built from sources under `C_Extensions/Editor/`.
 - The build pipeline usually generates both the compiled module and a `.pyi` stub for typing/autocomplete, then places them into `EditorExtensions/`.
 - If the native module cannot be imported, the editor should remain usable but may become noticeably slower on large maps.
 
 #### Adding custom widgets / panels (Widgets/)
 
 The editor UI is implemented in PyQt5 and lives under `Widgets/` (with reusable pieces under `Widgets/Utils/`). To add a new widget or panel:
 - Follow the existing naming/layout conventions (for example `W_*.py` for widgets and `WU_*.py` for utilities).
 - Keep the widget’s responsibilities narrow (one panel = one concern).
 - Integrate the panel by wiring it into the main window (menus, toolbars, dock areas, or existing panels) rather than creating standalone windows when possible.
 
 When adding panels, focus on:
 - Input/selection flow (how users enter the panel and how it interacts with the current map/tileset/script selection).
 - Data ownership (what data the panel edits, and when it writes back to disk).
 - Refresh and invalidation (what needs reloading after edits so the preview/runtime stays in sync).
 
 ### Extending the Runtime
 
 Ludork’s runtime is a Python-driven engine with native acceleration. The reference implementation used for projects lives under `Sample/Engine/`, and it is paired with a set of C/C++ modules built with pybind11.
 
 #### C Extensions in C_Extensions/
 
 Native modules are organized by responsibility:
 - `C_Extensions/Editor/` for editor-side acceleration (for example, fast tilemap rendering in the editor).
 - `C_Extensions/GamePlay/` for gameplay workloads (for example, pathfinding, passability cache rebuilds, particle updates).
 - `C_Extensions/Graphics/` for graphics-side helpers (tilemap graphics, map graphics, animation compression, etc.).
 - `C_Extensions/Utils/` for general acceleration utilities.
 
 Each module typically:
 - Exposes a Python-facing API (importable module) that the editor/runtime calls.
 - Provides a `.pyi` stub generated via `pybind11-stubgen` for IDE support.
 
 #### Adding new C++ modules via pybind11 (high-level)
 
 If you want to add a new native module, keep the integration lightweight and consistent with the existing modules:
 - Create a new folder under `C_Extensions/` following the existing structure (`include/`, `src/`, `CMakeLists.txt`, `main.cpp`, `process.py`).
 - Use `pybind11_add_module(...)` in `CMakeLists.txt` and expose a minimal, stable Python API in `main.cpp`.
 - Add a post-build step similar to the existing `process.py` scripts: generate the `.pyi` stub and move both the compiled module and stub into the expected target folder (either `EditorExtensions/` or under `Sample/Engine/`).
 
 Keep module scope small. Treat the C++ layer as an accelerator for well-defined hotspots rather than as the primary place to implement gameplay logic.
 
 ### Performance Tips
 
 #### Tilemap optimization
 
 - Prefer the native tilemap renderer in the editor when working with large maps; if it fails to load, expect slower redraws when painting tiles or toggling layers.
 - Reduce avoidable redraw work: hide layers you are not editing, and avoid extremely large “all-in-one” maps when a scene can be split.
 - Keep tileset images reasonable in resolution; large textures increase both VRAM pressure and conversion costs in the editor.
 
 #### Particle optimization
 
 - Cap particle counts and emission rates; particle systems scale poorly if you create too many short-lived objects every frame.
 - Prefer reuse/pooling over frequent allocations where possible.
 - If you notice spikes, reduce per-frame work: simplify particle behaviors, shorten lifetimes, and limit layered blending.
 
 #### Reducing memory usage (psutil / pympler)
 
 Ludork includes optional runtime-side memory/FPS diagnostics. In debug mode, the runtime can report:
 - Process RSS memory (via `psutil`)
 - Rough object size estimates (via `pympler.asizeof`)
 
 Use these numbers as directional signals to find growth trends (for example, texture/audio caches that only grow). Typical ways to reduce memory footprint:
 - Avoid loading unused textures/audio into long-lived caches.
 - Prefer smaller textures and compressed audio formats where appropriate.
 - Release or reuse large scene objects instead of recreating them repeatedly.
 
 ### FAQ & Troubleshooting
 
 #### Common installation issues
 
 - Missing Python dependencies: verify your environment matches `requirements.txt` and install all required packages.
 - C extension build failures: ensure you have a working C/C++ toolchain, CMake, and that SFML sources are available for the build scripts that depend on them.
 - Platform-specific Qt issues: PyQt5 may require platform plugins; failures often show up as “Qt platform plugin” errors when launching.
 
 #### Editor won’t launch / crashes
 
 - Launch from a terminal to capture the full Python traceback and identify the first failing import.
 - If it fails while importing native modules, rebuild the corresponding C extension and confirm the compiled artifacts are in the correct target folder.
 - If it crashes only in the packaged app, compare against the packaging include list and confirm required Qt plugins and Python modules are bundled.
 
 #### Missing assets / incorrect paths
 
 - Confirm the Project’s folder structure is intact (`Data/`, `Assets/`, and related subfolders).
 - Verify configuration entries (for example, `Data/Configs/System.json`) reference existing files.
 - Watch for case mismatches in file names when moving Projects between systems.
 
 #### Runtime does not reflect editor changes
 
 - Make sure content is saved before previewing/running.
 - Restart the preview/run session if changes involve code-like data that is cached (for example, Common Functions or imported modules).
 - If you maintain custom loaders, make sure they invalidate caches on reload and re-read updated JSON/data files.
 
 #### Packaging problems (missing files, launch errors)
 
 - Ensure your packaged output includes the Project data and assets folders; missing `Sample/` (or the equivalent Project directory) is the most common cause of runtime failures.
 - If the packaged app launches but assets are missing, check whether assets were outside the expected Project folder and therefore not copied.
 - If the packaged app fails to launch, verify included modules (for example PyQt5 SVG support) and required Qt plugins are present in the build.
