# 06. Data, Localisation, Nodes, and Testing

This chapter explains how to maintain general data, localisation, common functions, and node flows in the editor, then verify the result with a test run.

## General Data

`General Data` maintains rules data that is not placed directly on a map. The current template includes:

- `Item`: keys, enemy book, floor teleporter, break-wall, clear-wall, break-lava, and break-ice items.
- `Equip`: equipment data.
- `Enemy`: enemy names, descriptions, stats, drops, and special effects.
- `State`: poison, weak, and other states plus their state events.
- `Class`: class or actor category data.
- `Special`: template-specific special markers.

Choose field types based on the data's purpose. Use `string` for text, `int` or `float` for numbers, `bool` for switches, `file` for asset references, and `list` or `dict` for multi-value content.

File fields select files under `Assets`, which keeps saved paths stable.

## General Data Events

General data members can also have event graphs. For example:

- Item `onUse`.
- State `onWalk` and `onHookTriggered`.
- Enemy `onDefeat` and `onEncounter`.

These events are useful for logic bound to the data itself. For example, item use effects belong on item data, and per-step state effects belong on state data.

## Common Functions

`Common Functions` are stored in `Data/CommonFunctions`. The current template includes `PrintTest`.

Common functions are useful for reusable node flows that can be called by blueprints, items, states, or other nodes. If a node flow is reused by several events, prefer making it a common function.

## Node Functions

Node functions are grouped in the node menu. Common template groups include:

- `System`
- `Scene`
- `Player`
- `State`
- `Movement`
- `Mota`
- `Math`
- `String`
- `Container`
- `Utils`

When editing a graph, first place nodes that provide the current scene, player, map, or target actor, then connect them to the action nodes.

Nodes with multiple execution results show multiple output branches. Nodes that wait for movement, transitions, dialogues, or selections pause the flow and continue from the matching branch when complete.

## Localisation

Project text can be written as locale keys such as `{KEY_Y}` or `{SLIME}`. These keys are maintained in `Data/Locale/Locale.xlsx` and exported to `Data/Locale/en_GB` and `Data/Locale/zh_CN`.

When changing text, add or edit the ID and language columns in the locale table first, then use the editor's locale export action to regenerate the language files.

Keep the `{ID}` format when referencing localised text from general data, blueprints, or node parameters.

## Test Run

After saving the project, use `Test Game` to run the current project. During testing, check that:

- A new game starts from the expected map and position.
- Map tiles, autotiles, lights, and actors render correctly.
- Collision, overlap, doors, stairs, and item events trigger.
- Dialogues, selections, shops, weather, flashes, shakes, and transitions follow the node flow.
- Language, volume, window skin, cursor, and title music come from the current configuration.

If the test still shows old content, first confirm that the related maps, blueprints, general data, configuration, and locale table have been saved or exported.
