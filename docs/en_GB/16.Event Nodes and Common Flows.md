# 16. Event Nodes and Common Flows

The Sample game uses visual event graphs for blueprint behaviour and common reusable flows. This page describes both the editor usage and the runtime functions that those graphs can call.

## Blueprint Event Pages

Blueprints can define pages such as creation, update, collision, overlap, removal, encounter, and defeat. The exact pages depend on the actor class and blueprint data.

For logic that runs every frame or at a fixed timestep, prefer `onTick`, `onLateTick`, `onFixedTick`, and the matching event pages. Avoid putting gameplay logic in derived-class `update`, `lateUpdate`, or `fixedUpdate` methods; those entry points are normally owned by base classes for tick dispatch, animation updates, child synchronisation, or other shared state, and a missing `super()` call can cause subtle breakage.

Common uses include:

- doors checking keys or conditions;
- stairs transferring maps;
- items changing inventory or terrain;
- enemies resolving battle results;
- NPCs showing dialogue or selections.

## Common Flows

`Data/CommonFunctions` contains reusable graphs. Open `Common Functions` from the database menu, or press `F8`. The default Sample project includes `PrintTest.dat`.

Use common flows for shared tips, common rewards, repeated map logic, or debugging flows that should be callable from several blueprints.

## Blueprint Decorators

Blueprint functions and classes are exposed to the editor through decorators in `Sample/Engine/Decorators.py`. `@Meta` controls display names and specialised parameter editors; `@ExecSplit`, `@Latent`, and `@ReturnType` define execution and data pins. See `19.Blueprint Decorators` for the full reference.

## Node Function Groups

**Container**

Dictionary, list, `ForLoop`, and `ForEach` operations.

**Math and String**

Numeric, vector, comparison, boolean, random, and text helpers.

**Movement**

Movement routes and automatic pathing.

**Player**

Inventory, equipment, player values, gold, EXP, and player lookup.

**Scene and Mota**

Map transfer, terrain edits, shops, enemy book, floor teleport, and tower region helpers.

**State**

Battler states, damage, healing, and event arguments.

**System**

Audio, video, transition, flash, shake, and weather.

**Utils**

Local values, game variables, animations, common function calls, return-to-title, and print output.

## Loop Nodes

`Container.ForLoop` and `Container.ForEach` provide synchronous loop control in visual event graphs.

`ForLoop` loops from `firstIndex` to `lastIndex` inclusively, using `step`. Each iteration continues through `LoopBody` and outputs the current `index`; after the last iteration, execution continues through `Completed`.

`ForEach` iterates over a list. Each iteration continues through `LoopBody` and outputs `element` and `index`; after the list is exhausted, execution continues through `Completed`.

Loop bodies currently must be synchronous. Do not place latent nodes such as dialogue, selections, timers, transitions, path movement, or shop windows inside `LoopBody`; the runtime raises an error instead of trying to resume a suspended loop.

## Calling Common Functions

Game code can invoke a common function through the utility node layer:

```python
from Source.NodeFunctions.Utils import RunCommonFunction

RunCommonFunction("PrintTest")
```
