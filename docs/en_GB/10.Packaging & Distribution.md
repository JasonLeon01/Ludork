 # Ludork: 2D RPG Toolkit - Help Manual
 ## Packaging & Distribution

 This chapter explains how to package a **Project** into a distributable build. Ludork’s editor includes a packaging dialog that assembles the runtime entry, data, and assets into an output folder (and optionally produces an executable/app).

 ### Packaging Goals

 When you package a Project, you want to:
- Produce a standalone folder or app that runs without the editor.
- Include everything needed at runtime: `Assets/`, `Data/`, and the bootstrap entry (`Entry.py`, `Main.ini`).
- Include your Project runtime code when needed (for example `Engine/`, `Global/`, `Source/`).

 ### Packaging from the Editor

 Typical flow:
 - Save your Project (maps, configs, Common Functions, blueprints, etc.).
 - Open the **Pack** dialog and choose an output directory.
 - Pick a packaging mode and start the build.

 In the current Sample implementation there are two modes:
 - **Simple copy**: copies the required Project folders/files into the output directory. It copies `Assets/`, `Data/`, `Engine/`, `Global/`, `Source/`, plus `Entry.py`, `Main.ini` (and `Main.exe` if present). Good for quick sharing.
 - **Nuitka (Full)**: compiles `Entry.py` into an executable/app via Nuitka and bundles `Assets/`, `Data/`, and `Main.ini`. On Windows/macOS it checks for Python 3.12.0 to keep builds consistent.

Note: in the current implementation, Simple copy does not support video playback. Use the Nuitka mode if your Project needs video features.

 Platform note:
 - **macOS**: the current implementation forces the Nuitka mode (Simple is disabled).

 ### Output and Contents

 After packaging, you can expect an output folder or application that includes:
- Project assets: `Assets/` (images, audio, shaders, fonts, etc.).
- Project data: `Data/` (maps, tilesets, configs, Common Functions, exported locale, etc.).
- Bootstrap entry and startup settings: `Entry.py` + `Main.ini` (or the compiled output).
- Runtime code: Simple copy includes `Engine/`/`Global/`/`Source/` as folders; Nuitka bundles them via `--follow-imports`.

 You can test the packaged build by:
- Running the produced executable/app (or, for Simple copy, running `Entry.py` from the output directory with Python).
- Confirming maps, audio, fonts, and shader effects match what you saw in the editor.

 If anything is missing, check:
- That `Main.ini`’s `script` points to an existing entry script (the default is `Entry.py`).
- That `Assets/` and `Data/` are complete and that referenced files still exist (keep relative paths and folder structure consistent).

 ### Iterating on Builds

 Packaging is not a one‑time step. A typical loop is:
 - Edit content in the editor and test using the built‑in preview.
 - When you reach a stable state, create a packaged build.
 - Test the packaged build on the target platform.
 - Return to the editor to fix issues or polish content, then create a new build.

 Keeping your Project structure consistent and assets well‑organized helps the packaging process run smoothly and reduces surprises in the final build.
