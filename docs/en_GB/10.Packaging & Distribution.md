 # Ludork: 2D RPG Toolkit - Help Manual
 ## Packaging & Distribution

 This chapter explains how to package your game project into a distributable build. Ludork supports three target platforms: **Windows**, **macOS (Apple Silicon)**, and **iOS**. The editor provides a built-in packaging dialog that guides you through the entire process.

 ### Packaging Goals

 When you package a project, you want to:
- Produce a standalone executable or app that players can run without the editor.
- Include everything needed: assets (`Assets/`), data (`Data/`), and the bootstrap entry with startup settings.
- Bundle all runtime code so that players do not need to install Python or any other dependencies.

 ### Packaging from the Editor

 Typical flow:
 - Save your Project (maps, configs, Common Functions, blueprints, etc.).
 - Open the **Pack** dialog and choose an output directory.
 - Pick the target platform in the platform selection dialog, then start the build.

 The editor shows available target platforms based on the current operating system:
- On **Windows**: **Windows (win32)**.
- On **macOS**: **macOS (Apple Silicon)** or **iOS**.

 ### Windows Packaging

 **Prerequisites**:
- Python 3.12.0 — if not installed the editor will prompt you to download it from python.org.
- MSVC toolchain — if missing the editor will prompt you to download Visual C++ Build Tools.
- Nuitka — auto-installed by the editor if not present.

 **What happens**:

 The editor uses Nuitka to compile your game into a standalone Windows executable. Your `Assets/`, `Data/`, `Main.ini`, and all runtime engine code are automatically included. If `Assets/System/icon.ico` exists it is set as the application icon.

 **Output**: `Main.exe` (plus dependency files) in the output directory. Double-click to run your game.

 ### macOS (Apple Silicon) Packaging

 **Prerequisites**:
- Python 3.12.0 — if not installed the editor will prompt you to download it.
- Xcode toolchain — if missing the editor will prompt you to download Xcode.
- Nuitka — auto-installed by the editor if not present.

 **What happens**:

 The editor uses Nuitka to compile your game into a macOS `.app` bundle. `Assets/`, `Data/`, `Main.ini`, and all runtime engine code are automatically included. If `Assets/System/icon.icns` exists it is set as the application icon.

 **Output**: `Main.app` in the output directory. Double-click to run your game.

 ### iOS Packaging

 iOS packaging works differently from Windows/macOS — instead of producing a runnable app directly, it generates an **Xcode project** that you compile and sign in Xcode.

 **Prerequisites**:
- **macOS only**: the iOS packaging option is only available on macOS.
- Xcode toolchain (including the iOS SDK) — if missing the editor will prompt you to download Xcode.

 **What happens**:

 The editor generates a complete Xcode project that already includes references to your game entry, runtime environment, and native extensions. Icon files found in `Assets/System/` (`icon.icns`, `icon.ico`, or `icon.png`) are automatically converted into the iOS app icon.

 **Output**: `<project_dir>/build/<project_name>/`, containing:
- `<project_name>.xcodeproj/` — an Xcode project ready to open.
- `<project_name>/` — application source files and icon.

 After generation, open the `.xcodeproj` in Xcode, connect an iOS device, and build to deploy to the device.

 **Note**: iOS is limited to OpenGL ES 1 and cannot use shaders. Visual effects that rely on shaders (such as transitions, colour grading, and post-processing filters) will be unavailable or may look different compared to the desktop build. It is recommended to test the visual output on an iOS device and adjust or replace these effects as needed.

 ### Output and Contents

 Regardless of platform, the packaged build includes:
- Project assets: `Assets/` (images, audio, shaders, fonts, etc.).
- Project data: `Data/` (maps, tilesets, configs, Common Functions, exported locale, etc.).
- Bootstrap entry and startup settings: `Entry.py` + `Main.ini` (or the compiled output).
- Runtime engine code (bundled automatically; no manual steps required).

 You can test the packaged build by:
- **Windows**: running `Main.exe` from the output directory.
- **macOS**: double-clicking `Main.app`.
- **iOS**: opening the generated `.xcodeproj` in Xcode and running on a connected device.

 If anything is missing, check:
- That `Main.ini`'s `script` points to an existing entry script (the default is `Entry.py`).
- That `Assets/` and `Data/` are complete and that referenced files still exist (keep relative paths and folder structure consistent).

 ### Application Icon

 Ludork automatically sets the application icon for each platform. Place your icon file in your project's `Assets/System/` directory:
- **Windows**: `icon.ico`
- **macOS**: `icon.icns`
- **iOS**: `icon.icns`, `icon.ico`, or `icon.png` (searched in this priority order)

 If no icon file is provided the packaged build will use a default icon.

 ### Iterating on Builds

 Packaging is not a one-time step. A typical loop is:
 - Edit content in the editor and test using the built-in preview.
 - When you reach a stable state, create a packaged build.
 - Test the packaged build on the target platform.
 - Return to the editor to fix issues or polish content, then create a new build.

 Keeping your Project structure consistent and assets well-organised helps the packaging process run smoothly and reduces surprises in the final build.
