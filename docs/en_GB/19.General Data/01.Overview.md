# 01. Overview

General Data stores reusable RPG records—classes, players, enemies, equipment, items, battle specials, and states. Blueprints reference these records by `ID`; names, stats, icons, and event logic are resolved from General Data at runtime.

## Opening the Editor

Open `General Data` from the **Database** menu, or press `F10`. Each tab corresponds to one file under `/Data/General/` (for example `Enemy.dat`).

## File Layout

Each `.dat` file contains:

- **params**: the field schema shared by every member in that type (name, type, default value).
- **members**: one record per key; the key is the record `ID`.
- **linkedType** (optional): links the tab to a `Source/Infos/*.py` class so members can host blueprint event graphs.

Member records may also contain `_graph`, which stores per-member event node graphs. This is managed through **Edit Blueprint** in the editor and is not shown as a normal property field.

## Editor Layout

**Left panel**

- Lists all member IDs for the active tab.
- `+` adds a member; right-click a member to change ID, remove it, or open its blueprint graph when a linked type is set.

**Right panel**

- Shows the selected member's fields.
- `ID` is read-only; rename it through the member context menu.
- `+` at the bottom adds a new field to the type schema (applied to every member).
- `-` beside a field removes that field from the schema and from all members.

**Tab bar**

- Right-click a tab to create a new data type, rename the type, set or clear its linked type, or delete the type.

## Field Types

| Type | Editor control | Notes |
|------|----------------|-------|
| `string` | Single-line text | Often holds locale keys such as `{KEY_Y}`. |
| `int` | Integer spin box | Battle stats, prices, and counts. |
| `float` | Decimal spin box | Fractional values. |
| `bool` | Checkbox | Flags such as `usable` or `stackable`. |
| `file` | Browse button | Path relative to `/Assets/`. |
| `list` | Row list editor | Ordered string entries, for example drop IDs. |
| `dict` | Key-value rows | String keys and string values; battle code may evaluate values. |

When adding a field, set a default value for new members. Field names should match the attribute names your game code reads (for example `icon`, `price`, `attrPlus`).

## Linked Types and Blueprint Events

A linked type (such as `EnemyInfo` or `ItemInfo`) connects the tab to a class under `Source/Infos/`. The editor discovers `@RegisterEvent` methods on that class and exposes **Edit Blueprint** for each member.

Event graphs are stored inside the member's `_graph` data. Use them for data-level logic (`onUse`, `onDefeat`, `onWalk`, and so on) without duplicating a map actor blueprint.

## Linking Map Blueprints

For enemies, equips, and items, the map blueprint's `attrs.ID` must match the General Data member key. The blueprint controls map presentation (texture, collision, placement); General Data controls stats, text, icons, and info-layer events.

Create the General Data record first, then point the blueprint at the same `ID`.

## Localisation

`name` and `desc` fields usually store locale keys in `{BRACES}` rather than raw display text. Fill matching keys in the locale workbook and export before testing.

## Save and Test

Use **Save** after editing General Data. Start **Test Game** and trigger the record through battle, inventory, a door that consumes a key, or another path that reads the changed data.

## Default Sample Tabs

The Sample project ships seven tabs (listed in filename order):

- `Class` — player class and equipment slots.
- `Enemy` — enemy stats, drops, and battle specials.
- `Equip` — wearable gear and stat bonuses.
- `Item` — keys, tools, and usable items.
- `Player` — party member identity, class assignment, and battle stats.
- `Special` — reusable battle ability definitions.
- `State` — battler status effects.
