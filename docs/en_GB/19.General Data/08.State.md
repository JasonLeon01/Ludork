# 08. State

The **State** tab maps to `/Data/General/State.dat`. It is linked to `StateInfo` and supports `onWalk` and `onHookTriggered` blueprint events.

## Fields

**name**, **desc**

Locale keys for UI and tooltips.

**icon**

Optional path relative to `/Assets/` for status display.

**stackable**

When `true`, applying the same state again increases stack count instead of replacing the instance. `Poisoned` in the Sample set is stackable; `Weak` is not.

## Blueprint Events

**onWalk**

Called each time the affected battler takes a step on the map.

**onHookTriggered**

Called when game code explicitly triggers the state's hook on the host battler.

Combat damage no longer routes through state blueprint events; use these for overworld or custom scripted behaviour.

## Runtime Application

Game and node code apply states by ID through `Battler.addState`. The runtime builds a `StateInfo` from the matching General Data member and tracks stacks when `stackable` is enabled.

## Default Sample Members

| ID | Notes |
|----|-------|
| `Poisoned` | Stackable; increases damage taken per stack in battle. |
| `Weak` | Reduces effective ATK by `2 × stacks` while active. |

## Usage Notes

- Copy an existing state when adding a new one so `stackable` and graph slots are present.
- State IDs used in node functions or Python must match member keys here.
- Icon paths follow the same `/Assets/`-relative rules as items and equips.
